# API v0.2 Requirements From Expert Review

## Purpose

This is the required delta from API v0.1 to v0.2 after expert review. Use it
after the Thursday sync to update the API document with confirmed 慧誠 field
names.

## Scope Decision

API v0.2 must continue to support only:

```text
synthetic iMVS vital payload
-> structured questions
-> answer loop
-> staff_review_summary
```

After 多寶's two-phase workflow clarification, v0.2 should prefer this
optimized flow when 慧誠 UI supports it:

```text
Phase 1 pre-vital intake while iMVS is measuring
-> vitals-ready payload
-> Phase 2 vital-aware follow-up
-> staff_review_summary
```

Fallback remains the simpler post-measurement flow if 慧誠 cannot ask questions
during measurement.

It must not introduce:

- diagnosis;
- final triage / acuity level;
- treatment advice;
- emergency order;
- production HIS / EMR / FHIR writeback;
- real patient data;
- raw ASR audio.

## Required Additions

### Session

Add:

- `session_expires_at`
- `session_state`
- `last_question_id`

Allowed `session_state` values:

- `active`
- `summary_ready`
- `expired`
- `abandoned`
- `error`

### Retry And Idempotency

Add:

- `request_id`
- `idempotency_key`

For demo, these can be stable UUID-like strings. They prevent duplicate answer
submission from advancing the flow twice.

### Vital Payload Quality

Add per session:

- `measurement_timestamp`
- `device_id`

Add per vital field when possible:

- `measurement_status`: `measured`, `missing`, `failed`, `manual_entry`, or
  `not_available`
- `quality_flag`: `ok`, `needs_review`, `device_error`, `out_of_range_demo`, or
  `unknown`
- `missing_reason`

### Two-Phase Workflow

Add:

- `workflow_mode`: `parallel_measurement_intake` or `post_measurement_only`
- `measurement_state`: `not_started`, `in_progress`, `complete`, or `failed`
- `vitals_ready`: boolean
- `question_phase`: `pre_vital_intake`, `post_vital_followup`, or `summary`
- `phase_reason`: short explanation for why the current question belongs in
  the current phase

Recommended additional endpoint for optimized flow:

```http
POST /api/triage-demo/sessions/{session_key}/vitals
```

Use it when Phase 1 starts before vital measurement is complete. The original
two-endpoint flow is still acceptable as a June fallback if the UI cannot safely
ask questions during measurement.

### Output Control

Add:

- `summary_visibility: "staff_only"`
- `handoff_required`
- `handoff_reason_codes`
- `staff_handoff_note`

Replace:

- `plan_support` -> `review_action`

Reason: `plan_support` may be read as a SOAP Plan / medical action. For June
demo, the output should only tell staff or clinicians to review the measured
vitals and reported symptoms.

### Error

Every error must include:

- `status: "error"`
- stable `error.code`
- human-readable `error.message`
- `retry_allowed`
- no fake summary

Recommended error codes:

- `missing_required_field`
- `unsupported_question_type`
- `invalid_session`
- `session_expired`
- `api_timeout`
- `measurement_quality_unavailable`

## Updated Staff Summary Shape

```json
{
  "status": "summary",
  "summary_visibility": "staff_only",
  "handoff_required": true,
  "handoff_reason_codes": [
    "reported_shortness_of_breath",
    "measured_lower_oxygen_saturation_demo",
    "measured_fever_demo"
  ],
  "staff_review_summary": {
    "format": "review_summary_demo",
    "subjective": [
      "Patient reports shortness of breath."
    ],
    "objective": [
      "Synthetic measured vitals include fever, increased respiratory rate, and lower oxygen saturation than expected for this demo scenario."
    ],
    "assessment_support": [
      "Respiratory complaint and measured vitals should be reviewed by staff."
    ],
    "review_action": [
      "Staff or clinician review required."
    ],
    "staff_handoff_note": "Please review measured vitals and reported symptoms.",
    "not_claimed": [
      "No diagnosis.",
      "No treatment advice.",
      "No final triage level.",
      "No production HIS/EMR/FHIR writeback."
    ]
  }
}
```

## First Respiratory Flow

The respiratory flow should be early-handoff, not full eight-question intake:

1. Phase 1 during measurement: chief complaint.
2. Phase 1 during measurement: dyspnea duration / patient-reported severity.
3. Vitals-ready update.
4. Phase 2 after vitals: chest pain / pressure.
5. Phase 2 after vitals: chronic lung disease / baseline oxygen / medication
   context.
6. Staff-review summary.

## Runtime Enforcement Checklist

Before sharing a demo after expert review, check:

- [ ] No `diagnosis` output field.
- [ ] No `plan_support` field.
- [ ] No final acuity / ESI output.
- [ ] No treatment advice.
- [ ] No safe-to-go-home wording.
- [ ] No FDA-cleared / 510(k)-ready claim.
- [ ] No real name, MRN, national ID, phone, address, raw audio, or real chart
      content in payloads or logs.
- [ ] API errors do not generate summary text.
- [ ] `summary_visibility` is `staff_only`.
- [ ] `handoff_required` is explicit for respiratory case.
- [ ] Phase 1 questions do not depend on vital values.
- [ ] Phase 2 questions are not emitted until `vitals_ready=true`.

Current implementation hook:

- `scripts/checks/smoke-demo.js` checks runtime strings, demo fixtures, and
  API example JSON for the expert-forbidden high-risk phrases and
  `plan_support`.
