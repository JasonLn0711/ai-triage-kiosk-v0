# 會後可整理出的明確結論

## 1. 六月 demo 流程收斂

最終會議傾向先採用比較保守的 **post-measurement-only flow**：

```text
iMVS 先完成 Vital Sign 量測
-> 呼叫 NYCU API，上傳 vital payload
-> NYCU 回傳 session_key + 第一題
-> iMVS 逐題送回答
-> NYCU 回下一題
-> 最後回 staff_review_summary / triage 類結果頁
```

原本討論的 two-phase flow（邊量測邊問 Phase 1 問題）仍是合理方向，但六月 demo 暫時不大幅改 iMVS 原本流程。

## 2. API flow 的會議共識

會議中 Ben / 工程端確認後，Jason 說明 Endpoint 1 與 Endpoint 3 可在六月 demo 合併：

```text
量完 Vital 後，iMVS 直接打一支 API：
vital payload -> NYCU
NYCU 回：
session_key + first question
```

後續用 Endpoint 2 反覆送 answer / 回 next question，直到 summary。

## 3. 題型

六月 demo 先支援：

- `single_choice`
- `multi_choice`
- 必要時再加 `scale`

UI 端希望每題選項可在同一頁完整顯示，避免拖曳和滑動造成操作負擔。

## 4. Voice input

會議中確認：六月 demo 先不使用語音輸入。原因包含：

- demo 電腦目前沒有麥克風
- ASR 會增加硬體、延遲、錯誤與隱私風險
- 目前 touch / choice-based flow 已足夠展示主線能力

## 5. Remote API 與 local fallback

主要路徑：

```text
iMVS / demo UI -> NYCU server / lab backend API
```

備援路徑：

```text
Local scripted demo / fake demo system
```

Jason 會準備一套 local fallback，避免現場 API 或網路出問題時 demo 中斷。

## 6. Triage / staff summary 的定位仍需再釐清

會議中 Johnny 提出關鍵問題：

```text
如果輸出只是事實整理，那 triage 在哪裡？
```

會議也同時確認，只要有判斷行為，就會碰到醫療責任與產品定位問題。因此六月 demo 的 safer positioning 應該維持：

```text
vital-aware intake support
staff_review_summary
human review
demo-only / synthetic case
```

避免宣稱：

```text
AI diagnosis
final triage level
treatment advice
production HIS/EMR writeback
```

## 7. Demo case 要能演得出來

Johnny 強調 demo 腳本要符合現場人體可控條件：

- 心率可以靠跑步、深蹲提高
- 體溫可以用熱水等方式模擬
- 血氧很難控制
- 血壓也不容易精準控制
- 最好設計健康組 / 不健康組對照，讓 AI engine 的差異更有感

因此 respiratory low-SpO2 case 雖然醫療展示性強，但現場演示可控性較差；tachycardia / high heart-rate case 可能比較容易演。
